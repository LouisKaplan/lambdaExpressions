Though it might not make sense at first, giving our expression variables the type interface solves a lot problems.

One major issue that comes with adding new features is compatibility. If a new variable type is introduced, code has to be rewritten so that the new type can be used in already-established places. By simply re-using an existing type, a lot of work was saved.


Remember this example?

doubleNumberFunction = (int a) -> a *2;


Without a variable type, it's pretty useless. However, that variable doesn't represent the int that it will eventually return, it represents the process.

OneNumber doubleNumberFunction = (int a) -> a *2;

Now it's a OneNumber. What is OneNumber? An interface:


interface OneNumber{
int modifyOneNumber(int x);
}

OneNumber is a Functional Interface, which essentially means it has exactly one abstract method. It can contain other methods, in theory, but exactly one must be abstract.

Our Lambda Expression matches up with that interface, and when we want to pass the interface as a parameter, we can do so by passing it as the type Interface.

Once an interface is in place, we can write any number of accompanying Lambda Expressions to go with it. All that matters is that they have the same parameters and return types.

Worth noting that the interface must continuously be compatible with all Lambda Functions that implement it. To this end, we have the option tag it with @FunctionalInterface. With this tag in place, your IDE will show errors when incompatibility occurs.

@FunctionalInterface
interface OneNumber{
int modifyOneNumber(int x);
}
